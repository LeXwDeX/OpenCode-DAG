DAG 工作流配置的创建、调整、校验命令。

所有对 DAG YAML 的读写操作必须通过本命令完成，遵守铁律 #22 和 #28（SYSTEM.yaml 不可改、配置必须合规）。

用户提供的操作或约束（honoring these）:
$ARGUMENTS

## How to investigate

根据用户请求的操作类型，执行相应逻辑：

### list 操作
列出所有可用的 DAG 工作流模板和已创建的配置。

### create 操作
1. 解析用户输入的参数（namespace/name, extends, overrides 等）
2. 验证目标路径是否在允许的范围内
3. 如果指定了 extends，验证基础模板是否存在
4. 合并基础模板和覆盖参数
5. 验证合并结果是否符合 SYSTEM.yaml
6. 写入目标路径
7. 如果 auto_validate=true，自动调用 dag_validate

### update 操作
1. 解析用户输入的修改内容
2. 验证现有配置是否存在
3. 应用修改
4. 验证修改结果是否合规
5. 写入更新后的配置

### validate 操作
1. 读取指定路径的 YAML 配置
2. 执行完整验证：
   - required_nodes 完整性
   - max_agents 限制
   - max_concurrency 限制
   - max_pushes 限制
   - shadow 路由终止检查
   - fallback 链循环检查
   - worktree 隔离策略检查
   - merge 策略兼容性检查
3. 返回验证报告（errors + warnings）

### preview 操作
1. 读取配置
2. 模拟 DAG 执行流程
3. 生成执行计划（节点顺序、依赖关系、并发度）
4. 输出预览结果

### diff 操作
1. 读取两个配置文件
2. 对比差异（节点、依赖、参数等）
3. 输出差异报告

### inherit 操作
1. 读取基础配置
2. 应用用户指定的覆盖参数
3. 生成新配置
4. 验证新配置合规性

### register hook 操作
1. 验证 hook 名称和定义
2. 追加到项目级配置的 registry.hooks

### register mcp 操作
1. 验证 mcp 名称和 URL
2. 追加到项目级配置的 registry.mcp_servers

## Questions

仅在缺少必要信息时询问用户：
- 未指定模板时，询问继承哪个基础模板
- 未指定覆盖参数时，询问需要修改的字段
- 验证失败时，询问是否自动修复或手动调整

使用 `question` tool 进行一次性批量询问（最多 5 个问题）。

## Writing rules

生成的 YAML 配置必须：
- 符合 SYSTEM.yaml 的铁律约束
- 使用清晰的命名（kebab-case）
- 包含必要的注释说明关键决策
- 遵循 YAML 格式规范（缩进、对齐）

排除：
- 违反铁律的配置（required_nodes、max_* 限制等）
- 未经验证的自定义参数
- 可能导致死锁或无限循环的依赖关系

## 非法 DAG 流程（绝对禁止创建）

以下流程在测试中被验证为绝对非法，任何情况下都不能创建：

### 1. 循环依赖（Circular Dependencies）

```yaml
# ❌ 非法：A → B → C → A 形成循环
branches:
  - name: main
    nodes:
      - name: step-a
        depends_on: [step-c]  # ❌ 循环：C 依赖 B，B 依赖 A，A 又依赖 C
      - name: step-b
        depends_on: [step-a]
      - name: step-c
        depends_on: [step-b]
```

**为什么非法**：DAG 必须是无环图，循环依赖会导致无限等待和死锁。

### 2. 非法状态转移（Invalid State Transitions）

```yaml
# ❌ 非法：尝试从终态逆转到运行态
branches:
  - name: main
    nodes:
      - name: step-a
        status: completed
        transition_to: running  # ❌ 非法：终态不可逆转
```

**非法转移示例**（以下转移绝对禁止）：
- `RUNNING → PENDING`（只能向前，不能回退）
- `COMPLETED → RUNNING`（终态不可逆转）
- `FAILED → RUNNING`（终态不可逆转）
- `CANCELLED → RUNNING`（终态不可逆转）

**为什么非法**：状态机铁律 #16 要求终态（COMPLETED/FAILED/CANCELLED）不可逆转。

### 3. 缺失必需节点（Missing Required Nodes）

```yaml
# ❌ 非法：缺少必需的 workflow 步骤
branches:
  - name: main
    nodes:
      - name: implement  # ❌ 缺少 skeleton, tdd, review
        task: "直接实现"
```

**为什么非法**：workflow 必须包含所有 required_nodes（skeleton, tdd, implementation, review），这是铁律 #22 的要求。

### 4. 引用不存在的依赖（Non-existent Dependencies）

```yaml
# ❌ 非法：引用不存在的节点
branches:
  - name: main
    nodes:
      - name: step-a
        task: "第一个步骤"
      - name: step-b
        depends_on: [step-x]  # ❌ 非法：step-x 不存在
        task: "依赖不存在的节点"
```

**为什么非法**：所有 depends_on 必须引用已定义的节点，这会导致调度器无法正确处理依赖关系。

### 5. 非法转移参数（Invalid Transition Parameters）

```yaml
# ❌ 非法：错误的 transition 名称
branches:
  - name: main
    nodes:
      - name: step-a
        from_status: pending
        to_status: running
        transition: invalid_transition  # ❌ 非法：不存在的 transition 类型
```

**合法 transition 类型**：
- `ENGINE_START`（pending → running）
- `ENGINE_PAUSE`（running → paused）
- `ENGINE_RESUME`（paused → running）
- `ENGINE_COMPLETE`（running → completed）
- `ENGINE_FAIL`（running → failed）
- `ENGINE_CANCEL`（任意 → cancelled）

**为什么非法**：必须使用预定义的 transition 类型，这确保状态转换的合法性和可追踪性。

## Validation rules

所有生成的配置必须通过以下验证：
1. required_nodes 完整性（skeleton, tdd, implementation, review）
2. 节点数量不超过 max_agents
3. 并发度不超过 max_concurrency
4. push 次数不超过 max_pushes
5. shadow 路由必须能终止
6. fallback 链不能有循环
7. worktree 隔离策略正确配置
8. merge 策略与 worktree 配置兼容

如果验证失败，输出详细的错误信息和修复建议。
