# DAG Workflow Orchestration API

OpenCode runs a **built-in DAG (Directed Acyclic Graph) workflow engine** as a background orchestrator. The engine is always active — every chat session can spawn DAG workflows that execute in the background while you continue other work.

## When to Trigger

Use DAG workflows when user intent matches these patterns (in any language):
- **Keywords**: `DAG工作流`, `工作流`, `DAG 工作流`, `DAGWORKFLOW`, `/DAGWORKFLOW`, `DAGWORKER`, orchestrated multi-step tasks, parallel-but-dependent stages, build pipelines, staged code-repair plans, or any "run these N steps with dependency order" request.
- **Signals**: ≥2 steps with explicit or implicit dependency ordering; fan-out/fan-in patterns; long-running multi-stage work where later stages depend on earlier outputs.

## Node & Workflow Statuses

### Node status: `pending | queued | running | completed | failed | skipped`
- `pending`  — waiting on unresolved dependencies.
- `queued`   — dependencies resolved, waiting for an execution slot (`running_count < max_concurrency`).
- `running`  — child agent session actively executing.
- `completed` / `failed` / `skipped` — terminal, immutable. `skipped` is for required-upstream failure or `condition` evaluated to `false`.

One permitted reversal: **retry**. When `retry.max_attempts > 1` and attempt fails before exhaustion, `failed → pending` (sole exception to the immutability rule).

### Workflow status: `pending | running | paused | completed | failed | cancelled`
- `completed / failed / cancelled` are terminal.
- `running ↔ paused` is a legal bidirectional transition:
  - `dagworker pause` moves a running workflow to `paused` (running nodes continue; no new nodes are scheduled).
  - `dagworker resume` moves a paused workflow back to `running` (ready nodes are scheduled again).
  - `dagworker step` stays in `paused` but executes exactly one ready node to completion/failure.
- A `paused` workflow can still be `cancelled`, but cannot be directly `completed` — it must resume or step first.

## The Tool

`dagworker` is a registered tool with 14 `action` values:

| action | purpose |
|---|---|
| `start` | Create and fork a workflow; returns `workflowId` immediately; execution runs in background. |
| `status` | Poll a workflow's real-time state — which nodes are `pending`/`running`/`completed`/`failed`, violations, duration. |
| `cancel` | Abort a running workflow. |
| `pause` | Pause a running workflow; running nodes continue and new nodes are not scheduled. |
| `resume` | Resume a paused workflow and schedule ready nodes. |
| `list` | See every workflow spawned in the lifetime of this process. |
| `replan` | Restructure a running workflow's tail; keeps frozen nodes immutable and writes one audit row. |
| `template_list` | List built-in workflow templates. |
| `template_show` | Render a built-in template with input without starting it. |
| `template_start` | Render and start a built-in template workflow. |
| `node_detail` | Inspect a single node: `dagworker { action: "node_detail", node_id: "<id>" }`. Returns status, output, error_info, retries, timestamps. |
| `history` | Query replan audit trail: `dagworker { action: "history", workflow: "<workflowId>", limit?: number }`. Returns ordered list of replan records. |
| `logs` | Query node execution logs: `dagworker { action: "logs", node_id: "<nodeId>", limit?: number }`. Returns structured log entries. |
| `step` | Execute exactly one ready node while a workflow remains paused. |

## Templates — where they live, how to use them

DAG templates are reusable `DAGConfig` factories. They are not stored in chat state.

| Question | Answer |
|---|---|
| Where are built-in DAG templates written? | In code: `packages/opencode/src/dag/integration/templates/index.ts`. Add a new template there, append its id to `DAG_TEMPLATE_IDS`, and register it in the `registry` map. |
| How does an agent discover available templates? | Call `dagworker` with `action: "template_list"`. It returns ids, names, descriptions, tags, and required agent names. |
| How does an agent read one template's generated DAGConfig? | Call `dagworker` with `action: "template_show"`, `template_id`, and optional JSON-stringified `template_input`. This renders the template into a `DAGConfig` and does not start a workflow. |
| How does an agent start from a template? | Call `dagworker` with `action: "template_start"`, `template_id`, and required JSON-stringified `template_input` containing at least `{ "goal": "..." }`. |
| Where can an agent see a DAG it created by hand? | A hand-written JSON `workflow` passed to `dagworker start` is a workflow instance, not a named template. After start, use `dagworker list`, `dagworker status`, `dagworker node_detail`, `dagworker history`, and `dagworker logs` to inspect it. It will not appear in `template_list` unless added to `templates/index.ts`. |

Minimal template commands:

```json
{ "action": "template_list" }
```

```json
{ "action": "template_show", "template_id": "product-doc-analysis", "template_input": "{\"goal\":\"audit DAG docs\",\"scope\":\"packages/opencode/src/dag\"}" }
```

```json
{ "action": "template_start", "template_id": "product-doc-analysis", "template_input": "{\"goal\":\"audit DAG docs\"}" }
```

## The Config Schema

`dagworker start` takes a single `workflow` argument: a **JSON string** of this shape:

```json
{
  "name": "string (required, human-readable)",
  "description": "string (optional)",
  "max_concurrency": 3,
  "timeout_ms": 600000,
  "nodes": [
    {
      "id": "build",
      "name": "Build",
      "description": "Compile the project",
      "dependencies": [],
      "required": true,
      "timeout_ms": 300000,
      "retry": { "max_attempts": 2, "delay_ms": 5000 },
      "worker_type": "string (subagent name, e.g. 'implement')",
      "worker_config": { "agent": "implement", "prompt": "your task prompt" },
      "condition": { "ref_node": "upstream_id", "op": "eq|ne|gt|lt|gte|lte|exists|not_exists", "value": "<compare against>" },
      "input_mapping": { "inputKey": { "ref_node": "upstream_id", "ref_path": "$.optional.subfield" } }
    }
  ]
}
```

### Worker Type Resolution

- `worker_type` 必须是目前已注册的 agent 名字；运行时通过 active Agent registry 解析。
- 默认 registry 通常含 `build` / `plan` / `general` / `explore`；`scout` 视 `experimental-scout` 开关而定。
- `implement` / `verify` / `review` / `archgate` / `patcher` 等均为自定义 agent，使用前需在 opencode.json 的 `agent.*` 字段注册。
- 运行时 canonical schema 是 JSON `DAGConfig` / `DAGNodeConfig`（来自 `src/dag/session/types.ts`）。

### Recognized `worker_config` keys

| Key | Type | Effect |
|---|---|---|
| `agent` | string | Agent-name override — takes precedence over `worker_type` when resolving the child session's agent. Usually `agent === worker_type`; only specify this when they diverge. |
| `prompt` | string | Task prompt handed to the child subagent. |
| `use_worktree` | `true` | Opt-in per-node git worktree isolation (default `false` — child shares the host worktree). |
| `subDagConfig` | `DAGConfig` | Reserved for `worker_type === "dag"` — triggers sub-DAG dispatch. Depth cap: 3. |

### `condition` (declarative skip/ready, WP-B1)

- Evaluates the output of `ref_node` (must be a member of this node's `dependencies[]`) using `op` against `value`.
- If the result is `false`, the node is marked `skipped` and propagates skip to its transitive downstream.
- **`required: true` nodes cannot declare `condition`** — schema rejects the config.

### `input_mapping` (declarative upstream data injection, WP-C1)

- `Record<inputKey, { ref_node, ref_path? }>` — for each input key, pull output (or a `ref_path` sub-field) of `ref_node` (must be in `dependencies`) and inject it into the node's prompt at spawn time.
- Safe defaults: missing / null output / path-not-found / beyond-deps → audit as `skipped` with a reason code; node is not auto-failed, just injected without that key.
- Large payloads are truncated per-entry and per-total (engine enforced).

## Async Execution, Live State

`dagworker start` **does not block**. It returns a `workflowId` and the workflow runs in a background fiber. While it runs:
- Use `dagworker status <workflowId>` to inspect progress (which nodes are `pending` / `running` / `completed` / `failed`, how many ready, how many violated).
- Use `dagworker list` for a global view across all workflows.

## The Companion Tool: node_complete

Each workflow node spawns a child agent session using the `worker_type` + `worker_config.prompt` you specify. Inside that child session, the agent sees an extra tool `node_complete` — the **only way** a node signals success or failure. If the subagent finishes its turn without calling `node_complete`, the engine marks the node `failed`.

## Replan — Restructure a Running Workflow's Tail

`dagworker action=replan patch={...}` where `patch` is a JSON string shaped like:

```json
{
  "workflow_id": "<required>",
  "add_nodes": [{ "id": "...", "name": "...", "dependencies": [...], "required": true, "worker_type": "...", "worker_config": {...} }],
  "remove_nodes": ["<namespaced ids of pending nodes>"],
  "update_nodes": [{ "node_id": "<namespaced>", "new_dependencies": [...] }],
  "new_max_concurrency": 5,
  "changed_by": "main-agent"
}
```

**Atomicity**: All DB writes (node deletes, node updates, node inserts, workflow.config update, one `dag_workflow_history` audit row) are applied in a single `Database.transaction`. The patch either fully applies and returns `ok: true` with a `history_id`, or fully rejects with `ok: false` and a `reason` — no partial outcomes.

# Part II: Agent Rules

## Hard invariants (enforced at `start` time)

| Constraint | Limit | Rejection |
|---|---|---|
| Node cap | ≤ 20 nodes per workflow | `max_nodes_exceeded` |
| Concurrency cap | `max_concurrency` ≤ 10 | `max_concurrency_exceeded` |
| Required nodes | `required: true` nodes cannot be skipped — failure triggers `required_node_failed` violation | violation recorded |
| No cycles | `dependencies[]` must form a DAG | cycle diagnostic at `start` time |

## Terminal state immutability

- Terminal node states (`completed`, `failed` when retries exhausted, `skipped`) accept no further transitions. Single exception: node-level `retry` (`failed → pending` before exhaustion).
- Terminal workflow states (`completed`, `failed`, `cancelled`) cannot be retried, replanned, or resumed in place. Start a new workflow for terminal failures.

## Explicit completion

- `node_complete` is the only node-result signal the engine recognises. It is visible to child agent sessions only, not to the orchestrating agent.
- If a subagent reaches end-of-turn without calling `node_complete`, the engine marks the node `failed` with `"node did not call node_complete tool"`.

## Replan constraints

| Rule | Effect |
|---|---|
| Frozen nodes (`queued` / `running` / `completed` / `failed` / `skipped`) | Reject any `remove_nodes` or `update_nodes` targeting them |
| Terminal workflow (`completed` / `failed` / `cancelled`) | Reject replan outright |
| `remove_nodes` / `update_nodes[].node_id` | Must use namespaced IDs (`${workflowId}::${cfg.id}`); un-namespaced IDs silently no-op (or throw "not found" for required nodes) |
| `required: true` nodes | Cannot be in `remove_nodes`, even if other required nodes completed |
| Cycles in patch | Rejected before any DB write |
| `max_concurrency` bump | Must remain ≤ 10 |
| Duplicate `cfg.id` in post-patch graph | RequiredNodesValidator rejects |

**Trigger keywords**: user says "replan", "revise the plan", "adjust the remaining stages", "add a verification step", or "drop the final stage".

## Failure investigation and recovery

1. Inspect runtime state before searching for on-disk JSON config: `list` → `status` → `node_detail` → `logs` → `history`.
2. Check workflow status BEFORE deciding to rebuild:
   - `running` or `paused` with completed + failed nodes → **partial recovery via replan**: keep completed nodes; `remove_nodes` for pending/downstream nodes that depended on the failure; add corrected replacement nodes; then `dagworker resume`.
   - Terminal (`failed` / `completed` / `cancelled`) → cannot reuse the workflow; reconstruct the config and `start` a new workflow.
   - replan only accepts **pending** nodes in `remove_nodes`/`update_nodes`. Frozen nodes (`failed`/`completed`/`skipped`/`running`/`queued`) are rejected.
   - `required: true` required-node failures cascade downstream and usually make the workflow terminal (required_node_failed violation + workflow → failed); in this case replan is blocked and a fresh workflow is necessary.
3. `failure_handler` runs only while `workflow.status === "running"` and the failing node is still `running`. It does not apply after terminal failure.
4. `failure_handler` fields: `enabled`, `agent?`, `diagnosis_timeout_ms?`, `on_diagnosis_timeout?`, `max_recoveries?`. The field `failure_handler.timeout_ms` does not exist; use `node.timeout_ms` for node execution timeout.
5. `dagworker start` validates `worker_type` BEFORE database writes — an unregistered `worker_type` causes immediate rejection with `"worker_type not found: <name>"`.
