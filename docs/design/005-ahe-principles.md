## 核心原则

**少废话，多精炼**

### 什么是 AHE？

AHE 代表：**A**ction（行动）, **H**igh-quality（高质量）, **E**fficiency（效率）

核心理念：
- **少说多做**：减少冗余文档，聚焦关键设计
- **精炼优先**：只记录不可省略的推理和决策
- **结果导向**：代码实现是最终的文档

## AHE 开发流程

### 1. 设计阶段
- 简短的设计文档（2-3 页）
- 聚焦核心架构和关键决策
- 明确依赖边界和接口定义
- 列出风险点和未解决问题

### 2. 实现阶段
- 先写测试（TDD 优先）
- 实现必须通过所有测试
- 代码本身就是最好的文档
- 注释只解释"为什么"，不解释"是什么"

### 3. 验证阶段
- 单元测试覆盖率 > 80%
- E2E 测试覆盖关键场景
- 性能测试验证约束
- LSP 零错误

### 4. 文档精简
- 实现完成后，设计文档只保留：
  - 架构概览（1-2 段）
  - 关键决策和理由
  - 已知限制和风险
- 删除实现细节
- 删除测试用例详情（测试代码即文档）

## AHE 文档结构模板

```markdown
# [模块名] 设计文档

## 概述（2-3 段）
模块的目的、核心能力、在系统中的角色

## 核心架构
- 目录结构
- 关键类和接口
- 依赖关系图

## 关键决策
决策 1: [决策内容]
- 理由: [为什么这样选择]
- 替代方案: [考虑过的其他方案]
- 风险: [潜在风险]

## 已知限制
- 限制 1
- 限制 2

## API 概览
[只列出关键接口签名，不展开实现]
```

## AHE 反模式

### ❌ 不应该做的事

1. **过度文档化**
   - 写长篇大论的设计文档
   - 详细记录每个函数的实现
   - 重复代码已经表达的内容

2. **测试即文档**
   - 测试用例不需要详细注释
   - 测试失败时会说明问题在哪里
   - 测试本身应该清晰到不需要注释

3. **实现细节**
   - 不要在文档中记录具体的算法实现
   - 不要在文档中记录性能优化细节
   - 这些应该在代码中体现

### ✅ 应该做的事

1. **架构决策**
   - 为什么选择这个架构？
   - 考虑了哪些替代方案？
   - 有什么风险和限制？

2. **接口设计**
   - 关键接口签名
   - 依赖边界
   - 扩展点

3. **已知限制**
   - 当前不支持什么？
   - 有什么性能瓶颈？
   - 有哪些假设条件？

## AHE 检查清单

在提交前检查：

- [ ] 设计文档是否精简（< 3 页）？
- [ ] 是否只记录了关键决策？
- [ ] 是否删除了实现细节？
- [ ] 代码实现是否清晰到不需要注释？
- [ ] 测试是否清晰到不需要注释？
- [ ] 是否只记录了"为什么"而不是"是什么"？
- [ ] 是否标记了风险和已知限制？

## AHE 指标

### 文档指标
- 设计文档: < 3 页
- 关键决策: 3-5 个
- 已知限制: 明确列出
- 代码注释密度: < 10%（只解释"为什么"）

### 代码指标
- 测试覆盖率: > 80%
- LSP 错误: 0
- 代码复杂度: < 10（Cyclomatic complexity）
- 重复代码: < 5%

## 应用示例

### 好的 AHE 文档

```markdown
# WorktreeManager 设计

## 概述
管理 Git worktree，为每个工作流提供隔离环境。

## 核心架构
- 使用 Git 原生命令
- 每个 worktree 独立目录
- 自动清理机制

## 关键决策
决策: 使用 Git worktree 而不是 Docker
- 理由: 更轻量，启动更快，适合本地开发
- 替代方案: Docker 容器（太重）
- 风险: 不支持跨平台（仅 Git 支持的系统）

## 已知限制
- 仅支持 Git 工作流
- 不支持跨平台（仅 Git 支持的系统）
- worktree 清理可能失败（需要 manual 清理）
```

## AHE 开发流程示例

### 阶段 1: 设计（1-2 天）
- 写 2-3 页设计文档
- 列出关键决策
- 识别风险点

### 阶段 2: 实现（3-5 天）
- 先写测试
- 实现功能
- 确保测试通过

### 阶段 3: 验证（1-2 天）
- 运行所有测试
- 性能测试
- LSP 检查

### 阶段 4: 精简（1 天）
- 精简设计文档
- 删除实现细节
- 确保代码清晰

## 总结

AHE 的核心是：**少说多做，结果导向**

- 代码是最好的文档
- 测试是最好的验证
- 设计文档只记录"为什么"
- 实现细节在代码中体现
