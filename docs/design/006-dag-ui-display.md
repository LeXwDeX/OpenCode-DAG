## 概述

在 OpenCode TUI 界面中显示 DAG 工作流的执行状态，让用户能够：
- 查看当前工作流的执行进度
- 查看 worker 的并发状态
- 了解 DAG 的总体进度

## 设计目标

### 核心需求
1. 显示当前工作流的 DAG 状态
2. 显示 worker 数量和并发情况
3. 显示 DAG 步骤进度（并发的多个节点算一个步骤）
4. 清晰直观的视觉反馈

### 非目标
- 不实现完整的 DAG 编辑器
- 不实现 DAG 流程的运行时修改
- 不实现 DAG 执行日志的详细视图

## 界面位置

### 方案 A: 右侧面板（推荐）

**位置**: 右侧 Modified Files 上方
- 新增一个 "DAG" 标签页，与 "Files"、"Modified" 并列
- 或者在 "Files" 标签页上方新增 "DAG" 标签

**优势**:
- 不干扰当前的终端操作区域
- 可以同时查看 DAG 和文件变更
- 符合现有的标签页布局

**劣势**:
- 需要扩展标签页系统
- 可能需要占用更多屏幕宽度

### 方案 B: 顶部状态栏

**位置**: 顶部状态栏右侧
- 显示简化的 DAG 进度：`[DAG 3/10]`

**优势**:
- 不占用额外空间
- 始终可见

**劣势**:
- 信息太少，无法显示详细信息
- 无法显示 worker 并发状态

### 方案 C: 悬浮窗口

**位置**: 右侧悬浮窗口
- 按 F2 键展开/折叠
- 或者自动显示在右侧（类似 git status）

**优势**:
- 可以显示详细信息
- 不影响现有布局
- 可以折叠，不占用空间

**劣势**:
- 需要额外的快捷键
- 可能遮挡终端内容

## 显示内容

### 核心信息（必须显示）

1. **DAG 进度**
   ```
   DAG [3/10] 步骤
   ├── 完成: 3
   ├── 运行中: 2
   └── 等待: 7
   ```

2. **Worker 状态**
   ```
   Workers: 10/10
   ├── worker-1: RUNNING (task-a)
   ├── worker-2: RUNNING (task-b)
   ├── worker-3: PENDING
   └── ...
   ```

3. **并发性**
   ```
   并发: 2/2
   [当前步骤] [等待步骤]
   ```

### 可选信息

1. **节点依赖图**
   ```
   A → B → D
   A → C → D
   └── D → E
   ```

2. **执行时间**
   ```
   总时间: 5m 30s
   当前步骤: 2m 15s
   ```

3. **错误信息**
   ```
   ❌ worker-3: FAILED (timeout)
   ```

## 关键决策

### 决策 1: 并发节点算一个步骤

**决策**: 并发的多个节点算一个步骤

**理由**:
- 并发节点通常是独立的，用户不关心执行顺序
- 简化进度显示，避免"3/4" 后又是 "4/4" 的困惑
- 用户关心的是"还有多少步骤没完成"，不是"还有多少节点没完成"

**风险**:
- 如果并发节点执行时间差异很大（一个 5 分钟，一个 5 秒），用户可能会困惑为什么"3/4" 后还有 "4/4"

### 决策 2: 实时更新 vs 定时刷新

**决策**: 实时更新（通过 events 驱动）

**理由**:
- 现有的 events 系统已经支持实时事件
- 不需要额外的定时刷新机制
- 用户体验更好，立即看到进度变化

**风险**:
- 高频更新可能导致 TUI 性能问题（需要节流）
- 需要实现节流机制（例如：最多每秒更新一次）

### 决策 3: 显示位置

**决策**: 方案 A（右侧标签页，推荐）

**理由**:
- 不干扰当前的终端操作
- 用户可以随时查看 DAG 状态
- 符合现有的布局风格

**风险**:
- 需要扩展标签页系统（可能需要修改 layout.ts）
- 可能需要占用额外的屏幕宽度

## 实现约束

### 技术约束

1. **依赖现有 events 系统**
   - 使用现有的 state machine events
   - 不需要新增事件类型

2. **TUI 框架限制**
   - 使用现有的 ink 框架
   - 不引入新的依赖

3. **性能约束**
   - 节流：最多每秒更新一次
   - 懒加载：只在用户查看时渲染

### 数据流

```
State Machine Events
        ↓
DAG Progress Tracker (聚合状态)
        ↓
DAG Display Component (渲染)
```

## 已知限制

1. **不支持自定义 DAG 模板**
   - 当前只支持预定义的 DAG 模板（skeleton、tdd、implement）
   - 不支持用户自定义 DAG 流程

2. **不支持运行时修改**
   - 用户不能在运行时修改 DAG 配置
   - 不能手动触发/跳过节点

3. **显示简化**
   - 并发节点算一个步骤
   - 不显示详细的执行日志
   - 不显示 worker 的详细状态

## 开放问题

### 问题 1: 如何显示节点失败？

**选项 A**: 显示错误消息
```
❌ worker-3: FAILED (timeout)
```

**选项 B**: 只显示数量
```
Failed: 1
```

**建议**: 选项 A，让用户能够看到具体的错误信息

### 问题 2: 如何处理并发节点的显示？

**选项 A**: 只显示数量
```
并发: 2/2
```

**选项 B**: 显示节点列表
```
并发:
├── worker-1: RUNNING (task-a)
└── worker-2: RUNNING (task-b)
```

**建议**: 选项 B，让用户能够看到具体的 worker 状态

### 问题 3: 是否需要显示 DAG 依赖图？

**选项 A**: 显示 ASCII DAG 图
```
A → B → D
A → C → D
```

**选项 B**: 不显示，只显示进度

**建议**: 暂时不显示，保持简洁。如果用户需要，可以手动查看 DAG 配置文件。

## 风险评估

1. **高风险**: TUI 性能问题
   - 缓解措施：实现节流机制
   - 备用方案：如果性能问题严重，可以考虑移除实时更新

2. **中风险**: 用户不理解"并发算一个步骤"
   - 缓解措施：提供清晰的帮助文档
   - 备用方案：如果用户反馈强烈，可以考虑显示详细的执行步骤

3. **低风险**: 显示信息过多，导致界面混乱
   - 缓解措施：保持简洁，只显示关键信息
   - 备用方案：可以折叠详细信息，只显示摘要
