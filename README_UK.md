# OpenCode (Покращене видання)

> **⚠️ Заява**: Цей проєкт є оптимізованим відгалуженням [opencode](https://github.com/sst/opencode), яке підтримується незалежним розробником на основі оригінальної версії. Цей проєкт **не має жодного відношення до офіційної команди OpenCode** і не перебуває з нею у жодних відносинах. Оригінальний проєкт було випущено командою opencode під ліцензією MIT; це відгалуження зберігає ліцензію MIT вищого рівня, додаючи кілька власних модулів (детальніше див. [NOTICE](./NOTICE)).

## Вступ

Цей проєкт є **переробленою та покращеною версією** офіційної версії opencode. Мета:

- 🔧 **Виправлення проблем з китайською мовою**: Усунення проблем сумісності з поділом китайських слів, обробкою символів CJK, повноширинними розділовими знаками, китайськими шляхами та сценаріями введення китайською IME у вищому рівні (детальніше див. [Список виправлень китайських функцій](./docs/localization/zh-hans-fixes.md))
- 🧩 **Надання робочого DAG-рушія робочих процесів**: Власна розробка [Harness-DAG-Workflow](./docs/harness-dag.md), що дозволяє LLM-агенту оркеструвати та запускати багатовузлові паралельні завдання в одному сеансі
- 🎯 **Збереження сумісності з вищим рівнем**: Увесь код вищого рівня під ліцензією MIT залишається без змін, не порушується оригінальна збірка, не забруднюється API вищого рівня

## Встановлення

```bash
# Пряме встановлення (YOLO)
curl -fsSL https://opencode.ai/install | bash

# Менеджери пакунків
npm i -g opencode-ai@latest        # також можна використовувати bun/pnpm/yarn
scoop install opencode             # Windows
choco install opencode             # Windows
brew install anomalyco/tap/opencode # macOS та Linux (рекомендовано, завжди актуальна версія)
brew install opencode              # macOS та Linux (офіційна формула brew, нижча частота оновлень)
sudo pacman -S opencode            # Arch Linux (Stable)
paru -S opencode-bin               # Arch Linux (Найновіший з AUR)
mise use -g opencode               # Будь-яка система
nix run nixpkgs#opencode           # або github:anomalyco/opencode для найновішої гілки розробки
```

> [!TIP]
> Перед встановленням видаліть старі версії до 0.1.x.

## Особливості цієї гілки

Ця гілка побудована на основі upstream opencode, **додано** або **значно посилено** такі можливості (деталі в розділах):

| Можливість | Короткий опис | Ліцензія |
|------|------|------|
| 🧩 DAG HARNESS — система оркестрації завдань | Дозволяє LLM-агентам оркеструвати багатовузлові паралельні робочі процеси в межах однієї сесії | AGPL-3.0 |
| 🪝 Реалізація HOOKS API — надмножина | Повна система хуків: 22 типи подій виконання × 5 типів виконання | MIT + вдосконалення в цій гілці |
| 🛡️ Легковаговий ізольований простір CODING | Двотрекове ізоляційне середовище: Sandbox + Worktree | MIT + вдосконалення в цій гілці |
| 🔧 Усунення помилок для китайської локалізації | Сегментація CJK / повноширинна пунктуація / сумісність з IME / шляхи з китайськими символами | MIT |

### 🧩 DAG HARNESS — система оркестрації завдань (власний модуль · AGPL-3.0)

Попередня назва — Harness-DAG-Workflow. Готовий до продуктивного використання **рушій робочих процесів на основі спрямованого ациклічного графа (DAG)**, який дозволяє LLM-агентам оркеструвати складні паралельні завдання в межах однієї сесії. Ключові можливості:

- **Автоматичне планування**: автоматичний запуск дочірніх агентів відповідно до залежностей між вузлами, паралельне виконання
- **Динамічне перепланування**: можливість перебудувати робочий процес під час виконання (додавання/видалення/зміна вузлів, регулювання максимальної кількості одночасних процесів)
- **Непорушна відповідність суворим правилам**: скінченний автомат, який не можна обійти, незворотні кінцеві стани, обов’язкова трансляція подій, пріоритет персистентності
- **Інтеграція зі Slash-командами**: `/dag-ctl` для керування виконанням, `/dag-worker` для конфігурації робочого процесу
- **Аудит із персистенцією**: схема з 6 таблицями SQLite, усі зміни станів можна відстежити

Повний архітектурний дизайн див. у [документації Harness-DAG-Workflow](./docs/harness-dag.md), посібник для розробників — в [AGENTS.md](./packages/opencode/src/dag/AGENTS.md).

> **Ліцензія**: Цей модуль ([`packages/opencode/src/dag/`](./packages/opencode/src/dag/), [`packages/opencode/src/tool/dag-worker.ts`](./packages/opencode/src/tool/dag-worker.ts), [`packages/opencode/src/tool/node_complete.ts`](./packages/opencode/src/tool/node_complete.ts) та пов’язані шаблони й документація) розповсюджується під ліцензією **GNU AGPL v3** — використання цього модуля зобов’язує робити відкритими всі модифікації. Деталі в [NOTICE](./NOTICE) та [packages/opencode/src/dag/LICENSE](./packages/opencode/src/dag/LICENSE)

### 🪝 Реалізація HOOKS API — надмножина

Ця гілка повністю зберігає та розширює систему хуків із upstream:

- **22 типи подій виконання**: `PreToolUse` / `PostToolUse` / `FileChanged` / `UserPromptSubmit` / `Stop` / `SessionStart` / `SessionEnd` / `PermissionRequest` / `PermissionDenied` / `SubagentStart` / `SubagentStop` / `TaskCreated` / `TaskCompleted` / `TeammateIdle` / `PreCompact` / `PostCompact` / `WorktreeCreate` / `WorktreeRemove` / `ConfigChange` / `InstructionsLoaded` / `StopFailure` / `PostToolUseFailure`
- **5 типів виконання**: `command` (shell) / `mcp` (інструмент MCP) / `http` (REST) / `prompt` (однокроковий LLM) / `agent` (багатокроковий LLM)
- **Протокол обміну повідомленнями у вигляді конвертів JSON через stdin/stdout**: повна документація протоколу доступна в [`hooks-reference.md`](./packages/opencode/src/session/prompt/hooks-reference.md)
- **Вдосконалення в цій гілці**: інтеграція із шиною подій робочого процесу DAG (події `workflow.*` / `node.*`) + підписка TUI + пересилання через HTTP API

### 🛡️ Легковаговий ізольований простір CODING

Ця гілка пропонує двотрекове ізоляційне середовище виконання, яке дозволяє агентам / користувачам безпечно запускати код у пісочниці, не забруднюючи реальний репозиторій:

| Рівень ізоляції | Механізм | Призначення |
|---------|------|------|
| **Sandbox** (легкий) | Тимчасова директорія + діагностика LSP + мультимовний інструментарій (Python/Node/TS/Go/Rust/C/C++) | Пробне виконання окремих файлів / невеликих експериментів |
| **Worktree** (важкий) | `git worktree` — незалежна гілка + окреме представлення файлової системи | Паралельне редагування кількома агентами, масштабний рефакторинг |

- 📦 **Інструмент Sandbox**: `packages/opencode/src/tool/sandbox.ts`, кожен sandbox має власний кеш залежностей (venv / node_modules), підтримує разовий режим `ephemeral` та асинхронні довготривалі завдання у фоновому режимі `background`
- 🌳 **Менеджер Worktree для DAG**: У робочому процесі DAG кожен паралельний вузол може автоматично отримувати окрему worktree-гілку; після завершення вузла його зміни вливаються в основну гілку через `git merge`

### 🔧 Усунення помилок для китайської локалізації (виправлені проблеми з upstream)

Для версій upstream виявлено та виправлено кілька проблем сумісності / зручності використання в контексті китайської мови, зокрема:

- **Китайська сегментація та підрахунок токенів**: некоректна обробка символів CJK у деяких токенізаторах
- **Сумісність із повноширинною пунктуацією**: толерантність до повноширинних двокрапок, лапок, дужок при розборі конфігурації
- **Обробка шляхів із китайськими символами**: коректна передача файлових шляхів, що містять пробіли та символи CJK, у хуках / sandbox
- **Сумісність із китайськими методами введення (IME)**: затримки введення та тремтіння курсору в TUI при активному вікні кандидатів IME

Конкретні записи про виправлення та регресійні тести див. у [`docs/localization/zh-hans-fixes.md`](./docs/localization/zh-hans-fixes.md).

> 💡 Якщо ви під час використання виявите інші проблеми з китайською локалізацією, будь ласка, надішліть кроки для відтворення в [розділі issues](./issues), я продовжу їх виправляти.

### 🔬 Інші дрібні виправлення (інтегровані виправлення з upstream)

Ця гілка повністю зберігає кілька невеликих виправлень зручності використання з upstream, які пройшли регресійне тестування:

| Проблема | Коміт із виправленням upstream | Область впливу |
|------|-----------------|----------|
| 📋 **Пошкодження вмісту при копіюванні-вставленні** — вставлений користувачем текст prompt обрізався або втрачав символи в TUI | `564cde393e` [#28156](https://github.com/anomalyco/opencode/pull/28156) | Зручність введення в TUI |
| 📐 **Макет не оновлювався після вставлення** — після вставлення довгого тексту висота поля prompt не розтягувалася автоматично, виникав візуальний ефект обрізання | `1124315267` [#28164](https://github.com/anomalyco/opencode/pull/28164) | Зручність введення в TUI |
| 📎 **Відсутність запасного варіанту при помилці запису в буфер обміну** — коли API `navigator.clipboard` викидав помилку (наприклад, у HTTP-середовищі), операція копіювання просто завершувалася з помилкою | `fe143df151` [#27993](https://github.com/anomalyco/opencode/pull/27993) | Кросбраузерна сумісність |
| 🎨 **Недостатній контраст кольору тексту на значку вставлення** — значок підсумку операції вставлення був важко читабельним у деяких темах | `e94aecaa08` [#27969](https://github.com/anomalyco/opencode/pull/27969) | Візуальний досвід у TUI |
| 📏 **Неправильна оцінка ширини символів CJK / східноазійських символів** — emoji, повноширинні символи, кандзі та інші східноазійські символи відображали ширину, що не відповідала фактичній, що призводило до зсуву курсору | Включено до системи виправлень сегментації CJK | Вирівнювання символів у TUI |
| ⌨️ **Тремтіння вікна кандидатів IME** — при активації китайського / японського методу введення курсор тремтів + спостерігалась затримка вставлення символів | Локальний патч-обхід | Зручність введення в TUI |

> Ця гілка не винаходить велосипед: проблеми, виправлені в upstream, синхронізуються при оновленні гілки `stable`; ця гілка переважно виправляє проблеми з китайською локалізацією / робочим процесом DAG, які ще не опрацьовані в upstream.

## Можливості, збережені з вищого рівня

Наступні можливості повністю походять із opencode вищого рівня (ліцензія MIT); це відгалуження не вносило функціональних змін:

### Настільний застосунок (BETA)

OpenCode також пропонує настільний застосунок. Його можна завантажити безпосередньо зі [сторінки випусків](https://github.com/anomalyco/opencode/releases) або [opencode.ai/download](https://opencode.ai/download).

| Платформа             | Файл для завантаження                     |
| --------------------- | ----------------------------------------- |
| macOS (Apple Silicon) | `opencode-desktop-darwin-aarch64.dmg`     |
| macOS (Intel)         | `opencode-desktop-darwin-x64.dmg`         |
| Windows               | `opencode-desktop-windows-x64.exe`        |
| Linux                 | `.deb`, `.rpm` або AppImage               |

```bash
# macOS (Homebrew Cask)
brew install --cask opencode-desktop
# Windows (Scoop)
scoop bucket add extras; scoop install extras/opencode-desktop
```

### Агенти

OpenCode містить два вбудовані агенти, між якими можна швидко перемикатися клавішею `Tab`:

- **build** — Режим за замовчуванням, має повні права доступу, підходить для розробки
- **plan** — Режим лише для читання, підходить для аналізу та дослідження коду
  - За замовчуванням відмовляє у зміні файлів
  - Запитує перед виконанням команд bash
  - Зручний для дослідження невідомих кодових баз або планування змін

Також включено під-агент **general** для складного пошуку та багатокрокових завдань; використовується внутрішньо, також можна викликати, ввівши `@general` у повідомленні.

Дізнайтеся більше про [Агентів](https://opencode.ai/docs/agents).

### Реалізація надмножини ClaudeCode Hooks API

Це відгалуження повністю зберігає екосистему Hooks API вищого рівня та 22 події запуску середовища виконання. Хуки реєструються в полі `hooks` конфігураційного файлу за назвою події, підтримують п'ять типів виконання: `command`, `mcp`, `http`, `prompt`, `agent` і комунікують через JSON-конверт stdin/stdout. Повний протокол див. у [`packages/opencode/src/session/prompt/hooks-reference.md`](./packages/opencode/src/session/prompt/hooks-reference.md).

Детальний перелік подій та таблицю типів виконання див. у [збереженому розділі функцій вищого рівня](./docs/readmes/upstream-features.md).

## Поширені запитання (FAQ)

### Чим це відрізняється від Claude Code?

За функціональністю дуже схоже. Ключові відмінності:

- 100% відкритий код
- Не прив'язаний до конкретного постачальника. Рекомендуються моделі [OpenCode Zen](https://opencode.ai/zen), але також можна використовувати з Claude, OpenAI, Google або навіть локальними моделями
- Вбудована підтримка LSP
- Фокус на термінальному інтерфейсі (TUI)
- Архітектура клієнт/сервер. Можна запускати на локальній машині та одночасно керувати віддалено з мобільного пристрою

### Чим це відрізняється від офіційної версії opencode?

- Додано рушій робочих процесів Harness-DAG-Workflow (AGPL-3.0)
- Постійне виправлення проблем сумісності у сценаріях використання китайської мови
- Довгострокове незалежне обслуговування, відокремлене від темпу вищого рівня
